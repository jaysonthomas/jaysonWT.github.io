RAII

Overview

Resource Acquisition Is Initialization. The goal is to have allocation and deallocation disappear from the surface level of the code thereby protecting resources such as a file stream, a network connection or a block of memory. This is achieved by:
  • Wrapping a management class around a resource (the resource is entirely managed within the class) while providing third-party access through well-defined interfaces only
  • Smart pointers

The main advantages

  • Helps to manage ownership and lifetime of dynamically allocated objects since class destructors perform resource clean-up tasks when the RAII object gets out of scope
  • Helps to implement encapsulation and information hiding easily since resource acquisition and release gets done within the same object

Acquiring and releasing resources

Example scenarios

  • Allocate/free memory
  • Open/close a file or network connection
  • Protect synchronisation primitives such as atomic operations, memory barriers, monitors or critical sections, which must be released to allow other threads to obtain them

General problems

  • The program might throw an exception during resource use and thus the point of release might never be reached

  • There might be several points where the resource could potentially be released, making it hard for a programmer to keep track of all eventualities

  • It is easy to forget to release a resource

RAII management class

There are 3 parts to an RAII class:
  • A resource is allocated in the constructor
  • A resource is de-allocated in the destructor
  • All instances of the RAII class are allocated on the stack to reliably control the lifetime via the object scope
